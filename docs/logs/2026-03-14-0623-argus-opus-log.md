## 06:25 — Design doc review: Project Instructions Inheritance

Read `docs/plans/project-instructions-inheritance.md` in full. This is a clean, well-reasoned design. My notes from a testing perspective:

### What I like
- **First-class `projects` table** is the right call. `sourceMetadata` was always a per-channel bag — trying to share it would have been painful. Clean relational model.
- **Two-field inheritance (project instructions + channel addendum)** avoids the concatenation awkwardness and the replacement rigidity. The layering order (`kit_briefing → project.instructions → channel.system_prompt → entity.systemPrompt`) is clear and testable.
- **Migration path** is conservative — nullable `project_id`, no FK constraint initially, existing channels unaffected. Good for incremental delivery.
- **Kit briefing deduplication** — moving CLAUDE.md/prompt_template out of kit briefing and into the project layer is the right separation. Kit briefing stays focused on orientation + capability warnings.

### Test implications for Argus
1. **Schema migration tests** — new `projects` table creation, `project_id` column on channels, data migration for existing imported channels with `sourceMetadata.claudeMd`
2. **Project CRUD** — create, read, update, delete projects; verify channel unlinking on delete
3. **Prompt assembly** — unit tests for `buildSystemPrompt()` with all layer combinations:
   - No project, no addendum (backward compat)
   - Project instructions only
   - Project instructions + channel addendum
   - Full stack: kit briefing + project + addendum + entity
   - Empty/whitespace handling at each layer
4. **Import integration** — claude.ai: `prompt_template` → `projects.instructions`; Claude Code: `CLAUDE.md` → `projects.instructions`; verify dedup (same cwd/project UUID → same project row)
5. **Kit briefing regression** — verify CLAUDE.md no longer injected via kit briefing when project layer handles it; verify MEMORY.md still injected via kit briefing
6. **UI tests** — project dropdown in channel settings, read-only project instructions display, overwrite warning when assigning channel to project

### Open questions I'd flag
- **Size limit (Q1):** Agree with 16K suggestion. CLAUDE.md files can be substantial. Should test truncation behavior and UI warning.
- **Project deletion (Q2):** Unlinking is the right default. Test that channels survive with their addendum intact.
- **Migration ordering:** The data migration (step 3 — creating projects from existing sourceMetadata) should probably be a separate migration function, not inline in the schema init. Makes it testable and rerunnable.

### Intersection with my current assignments
- **8¾d (Claude Code session browser):** No conflict. Session browser is UI + scanning, projects table is schema + prompt assembly. They're orthogonal.
- **Tests for 8¾a:** The design doc changes what 8¾a means. My test plan for project context injection should target the new `projects` table + `buildSystemPrompt()` layering, not the old `sourceMetadata`-only approach.

**Bottom line:** Design is solid, testable, and incremental. Ready to write tests as soon as Daedalus delivers schema + prompt assembly changes.
